Data Types

lPrism.datatype.vms.l3AddressData

The data type l3AddressData is defined by TOSCA-NFV and ETSI-NFV; its function is to inform about the configuration of the IP addresses within a subnet, such as whether the IP addresses will be assigned manually or automatically by the orchestrator.

The properties of l3AddressData are described in the following table:

Name Required Type Description
ipAddress-Assignment Yes Boolean Specifies whether the IP address will be assigned manually by the user or automatically by the orchestrator. The default value is True.
floatingIpActivated No Boolean
ipAddressType Yes String Specifies the type of address. Valid values: ipv4, ipv6
numberOfIp-Address No Integer Minimum number of IP addresses to assign.

Definition

                  
          lPrism.datatypes.vms.l3AddressData:
              properties: 
                  ipAddressAssignment:
                    type: Boolean
                    required: true   
                  floatingIpActivated:
                    type: Boolean
                    required: false
                  ipAddressType:
                    type: string
                    required: false
                    constraints:
                      - valid_values: [ipv4, ipv6]
                  numberOfIpAddress:
                    type: Integer
                    required: false

Example

                  
          l3AddressData:
              ipAddressAssignment: true
              floatingIpActivated: false
              ipAddressType: ipv4
              numberOfIpAddress: 1

lPrism.datatype.vms.addressData

This complex data type is used by TOSCA-NFV and ETSI-NFV to provide information about the address assigned to the connection point. The properties are listed below:

Name Required Type Description
addressType Yes String Describes the address assigned to the connection point. The assignment depends on the ipAddressAssignment attribute of l3AddressData.
l3AddressData No lPrism.datatype.vms.l3AddressData Provides information on the address assigned to the connection point.

Definition

          
  lPrism.datatypes.vms.addressData:
      properties:
          addressType:
              type: String
              required: true
              constraints:
              - valid_values: [mac_address, ip_address]
          l3AddressData:
              type: lPrism.datatypes.vms.L3AddressData
              required: false

Example

          
  addressData:
      addressType: IP address
      l3AddressData:
          ipAddressAssignment: true
          ipAddressType: IPv4

lPrism.datatype.vms.connectivityType

This complex data type describes the type of connection. Its structure is composed of the attributes described below:

Name Required Type Description
layerProtocol Yes String Identifies the connection protocol (ethernet, ipv4, ipv6, etc.)
flowPattern No String Identifies the connection mode (Line, Tree, Mesh)

Definition

          
  lPrism.datatypes.vms.connectivityType:
      properties:
          layerProtocol:
            type: string
            required: true
            constraints: 
            - valid_values: [ethernet, ipv4, ipv6]
          flowPattern:
            type: string
            required: true
            constraints:
            - valid_values: [Line, Tree, Mesh]

Example

          
  connectivityType:

      flowPattern: Line

lPrism.datatype.vms.virtualMemory

This data type is used by TOSCA-NFV to describe the properties of memory used by the virtual machine. The properties are listed in the table below:

Name Required Type Description
virtualMemSize Yes Integer Defines the amount of memory in MB assigned to the connection point (VMS). By default, it has the same value as minVirtualMemSize.
minVirtualMemSize Yes Integer Defines the minimum amount of memory that must be assigned to the connection point.

Definition

          
  lPrism.datatypes.vms.virtualMemory:
      properties:
          virtualMemSize:
            type: scalar-unit.size # Number
            required: true
          minVirtualMemSize:
            type: scalar-unit.size # Number
            required: true

Example

          
  virtualMemory:
      virtualMemSize: 1024
      minVirtualMemSize: 512

lPrism.datatype.vms.port

This data type describes an input or output port enabled on a VMS for multimedia streams. The properties are listed below:

Name Required Type Description

formatStream No List Multimedia stream formats used by the port.

Definition

          
  lPrism.datatypes.vms.port:
      properties:
          numPort:
              type: String
              required: true
          typeStream:
              type: String
              required: true

              - valid_values: [video, audio, image, text]

              type: list
              entrySchema:
                  type: String
              required: false        
          qualityStream:
              type: list
              entrySchema:
                  type: String

Capabilities Types

lPrism.capabilities.vms.metric

Metric is a type of particular capability used to monitor a virtualized element. This capacity is significant when a VMS sends reports on its operation. The Metric capability was modified to fit the V-PRISM architecture, but it still works as defined by TOSCA-NFV.

Name Required Type Description
state Yes Boolean --
listPorts No list ---

Definition

          
  lPrism.capabilities.vms.metric:
      properties:
          state:
            type: Boolean
            required: true
          listPorts:
            type: list
            entrySchema:
              type: lPrism.datatypes.vms.port
            required: false

Example

          
  metric:
    state: True
    listPorts: 
    -
      numPort: 15001
      typeStream: text

lPrism.capabilities.vms.virtualCompute

VirtualCompute encompasses the computational resources that can be assigned to a virtualized component. These resources can be configured in the VMS or preconfigured in the software image registry.

Name Required Type Description
VirtualMemory Yes lPrism.datatype.vms.virtualMemory Describes the minimum amount of RAM required by software images and the amount allocated to the virtualized component (VMS, VD).
VirtualCpu Yes lPrism.datatype.vms.virtualCpu Describes the minimum number of CPUs required by software images and the number allocated to the virtualized component (VMS, VD).
VirtualStorage Yes lPrism.datatype.vms.virtualStorage Describes the minimum amount of storage required by software images and the amount allocated to the virtualized component (VMS, VD).
VirtualGraphicsCard No lPrism.datatype.vms.virtualGraphicsCard Describes the minimum amount of graphic memory required (if required) by the software image and the amount allocated to the virtualized component.

Definition

          
  lPrism.capabilities.vms.virtualCompute:
      properties:
          VirtualMemory:
              type: lPrism.datatype.vms.virtualMemory
              required: true
          VirtualCpu:
              type: lPrism.datatype.vms.virtualCpu            
              required: true
          VirtualStorage:
              type: lPrism.datatype.vms.virtualStorage           
              required: true
          VirtualGraphicsCard:
              type: lPrism.datatype.vms.virtualGraphicsCard            
              required: false

Example

          
  virtualCompute:
      VirtualMemory:
          virtualMemSize: 1024
          minVirtualMemSize: 512
      VirtualCpu:
          numVirtualCpu: 2
          minNumVirtualCpu: 1
      VirtualStorage:
          virtualStorageSize: 10000
          minVirtualStorageSize: 5000
      VirtualGraphicsCard:
          virtualGraphicsCardSize: 1024
          minVirtualGraphicsCardSize: 512

Relationship Types

lPrism.relationship.vms.source

The structure of the source relationship type is detailed below.

Name Required Type Description
sourceName Yes String Describes from which component the media stream transmission starts. Must be identical to deviceName or vmsName.

formatType No String Describes the format of the media stream, e.g., video resolution (360p vs 4K).
networkBandwidth No lPrism.datatype.vms.networkBandwidth Defines the bandwidth assigned to this connection source for cross-host connections.

lPrism.nodes.vms.VDU.vms

A vms within a multimedia service chain is used for processing multimedia streams and generating output results, which may be sent to other VMS(s) or end user applications.

Name Required Type Description
vmsName Yes String The VMS name assigned by the developer of the multimedia service chain. It acts as an ID in a virtualLink.
vmsDescription No String Description of the VMS within the multimedia service chain.
configurableProperties No lPrism.data-type.vms.configurableProperties Describes the configurable properties of the VMS component, if applicable.
address No lPrism.data-type.vms.addressData Address of the virtualized component (IP, MAC).
network No String Describes the network the virtualized component belongs to.
ports No lPrism.data-type.vms.ports Describes the ports enabled by the virtualized component.
virtualCompute No lPrism.capabilities.vms.virtualCompute Describes the resource requirements like storage, memory, CPU, and graphics card.
monitoringParameter No lPrism.capabilities.vms.metric Describes monitoring parameters like CPU usage, memory consumption, etc.
virtualBinding No lPrism.capabilities.vms.virtualBindable Describes whether the virtualized component can function as a subscription-based service.
vmsType Yes lPrism.artifacts.vms.swImage Describes the software image used by the VMS.

Chain Type

lPrism.chain.vms.chainModel

L-PRISM represents a multimedia service chain as a language element named chainModel. A chainModel has three main components: Virtual Device (devices), Virtual Multimedia Sensors (vmss), and their connections (virtualLinks).

Name Required Type Description
chainName Yes String The name of the multimedia service chain.
chainDescription No String The description of a multimedia service chain.
devices Yes list List of virtual devices that compose the multimedia service chain.
vmss Yes list List of VMSs that compose the multimedia service chain.
virtualLinks Yes list List of virtual connections between virtual devices and VMSs.

Definition

                              
                      lPrism.chain.vms.chainModel:
                          properties:
                              chainName:
                                  type: String
                                  required: true
                              chainDescription:
                                  type: String
                                  required: false     
                              devices:
                                  type: list
                                  entrySchema:
                                      deviceName:
                                          type: String
                                          required: true
                                      deviceId:
                                          index:
                                          type: lPrism.artifacts.vms.device
                                          required: true
                                  required: true
                              vmss:
                                  type: list
                                  entrySchema:
                                      type: lPrism.nodes.vms.VDU.vms
                                  required: true
                              virtualLinks:
                                  type: list
                                  entrySchema:
                                      type: lPrism.nodes.vms.virtualLink
                                  required: true

Example

                              
                      chainModel:
                          chainName: VMS Chain
                          chainDescription: Video Flux
                          devices:
                          -
                            deviceName: example1 video ball
                            deviceId: 638e70b10a1fbd0026fccaf4   
                          vmss:
                          -
                            vmsName: vms1 example 1
                            vmsDescription: My VMS description
                            vmsType: 638e70b10a1fbd0026fccae4
                            startupParameters: # Null
                            host: 192.168.0.117 # IP
                            virtualCompute:
                              virtualMemory:
                                  virtualMemorySize: 1024 
                              virtualCPU:
                                  numVirtualCpu: 2
                              virtualStorage:
                                  viertualStorageSize: 10000

                          - #virtualLink 1
                              source:
                                sourceName: example1 video ball
                                sourceType: device
                                outputType: video 
                                formatType: # not define
                                networkBandwidth: # not define
                              destination:
                                destinationName: vms1 example 1 
                                destinationIp: #define for orchestrator
                                destinationPort: 
                                  numPort: 5000
                                  typeStream: video
                          - #virtualLink 2
                              source:
                                sourceName: vms1 example 1
                                sourceType: vms
                                outputType: video 
                                formatType: # not define
                                networkBandwidth: # not define
                              destination:
                                destinationName: 192.168.0.101 #My PC 
                                destinationIp: 192.168.0.101 #My IP
                                destinationPort:
                                  numPort: 20002 # my port
                                  typeStream: video
